## Deployment Modes

| Scenario | Pattern | Latency |
|----------|---------|---------|
| Single agent, local | Direct import / stdio MCP | under 1ms |
| Multiple agents, same machine | HTTP MCP server | 1-5ms |
| Multiple agents, distributed | HTTP MCP + S3 backend | ~50ms |
| Decentralized, no server | Multi-writer S3 | ~50ms (eventual) |
| Serverless (Lambda) | S3 per invocation | ~50ms |

## Write Modes

| Mode | Speed | Durability | Use case |
|------|-------|------------|----------|
| `immediate` (default) | 1x | Every write survives crash | Safety-first |
| `group` | ~12x | Crash loses up to 100ms | Batched workloads |
| `async` | ~50x | Crash loses all unflushed | High-throughput |

```bash
npx @backloghq/agentdb --http --write-mode group
npx @backloghq/agentdb --http --write-mode async
```

## Storage Modes

| Mode | Best for | Format | Notes |
|------|----------|--------|-------|
| `memory` (default) | < 10K records | In-memory with WAL | Fastest reads, all data in RAM |
| `disk` | 1M+ records | JSONL + Parquet | Point lookups via JSONL, column scans via Parquet |
| `auto` | Variable workloads | Starts memory, switches to disk | Switches when collection exceeds `diskThreshold` |

```typescript
// Explicit disk mode
const db = new AgentDB("./data", { storageMode: "disk" });

// Auto mode — promotes to disk at 10K records
const db = new AgentDB("./data", { storageMode: "auto", diskThreshold: 10_000 });
```

Disk mode works with both filesystem and S3 backends.

## S3 Backend

```bash
npx @backloghq/agentdb --backend s3 --bucket my-bucket --region us-east-1 --http
```

```typescript
import { AgentDB, loadS3Backend } from "@backloghq/agentdb";
const { S3Backend } = await loadS3Backend();
const db = new AgentDB("mydb", {
  backend: new S3Backend({ bucket: "my-bucket", region: "us-east-1" }),
});
```

Requires `@backloghq/opslog-s3` (optional dependency).

## Performance Knobs

Four knobs tune embedding throughput, disk concurrency, and Parquet layout. All four sit on `AgentDBOptions` (db-wide defaults) **and** on `CollectionOptions` (per-collection overrides) — per-collection wins, db-wide fills the gap.

```typescript
const db = new AgentDB("./data", {
  // Embedding provider call batching — records per provider call in
  // embedUnembedded / reembedAll. Default 256.
  embeddingBatchSize: 256,

  // Concurrent disk reads in materializeCandidates (search candidate fetch).
  // Default 16 for non-FS backends (S3); unbounded on local FS.
  diskConcurrency: 16,

  // Disk-mode LRU record cache size.
  cacheSize: 1000,

  // Parquet row-group size for column scans.
  rowGroupSize: 50_000,

  embeddings: { provider: "ollama" },
});

// Per-collection override
const events = await db.collection(defineSchema({
  name: "events",
  storageMode: "disk",
  embeddingBatchSize: 128,   // smaller batches for this collection
  diskConcurrency: 32,       // higher QPS workload
  fields: { /* ... */ },
}));
```

### S3 sizing guidance

`diskConcurrency: 16` is a deliberate default for S3-backed deployments — it keeps each prefix below per-prefix request rate limits while still saturating bandwidth for typical search workloads. Tune from there:

| Workload | `diskConcurrency` |
|----------|-------------------|
| Default (mixed) | `16` |
| Very high QPS, dedicated S3 prefix | `32` |
| Sharing a prefix with other tenants | `8` |
| Local filesystem | unbounded (flag ignored) |

`embeddingBatchSize` is provider-bound: OpenAI, Voyage, Cohere, and Gemini accept hundreds per call; Ollama is sequential and benefits less from large batches.

<Aside type="caution">
  When you pass a `defineSchema` result to `db.collection(schema, opts)`, `schema.collectionOptions` **completely replaces** the `opts` argument. Set `embeddingBatchSize` / `diskConcurrency` inside `defineSchema(...)` for schema-defined collections, or as db-wide defaults on `AgentDBOptions`.
</Aside>

## Read-only Mode

Open alongside a writer for dashboards and monitoring:

```typescript
const reader = new AgentDB("./data", { readOnly: true });
await reader.init();
const col = await reader.collection("tasks");
await col.tail(); // pick up latest writes
```
