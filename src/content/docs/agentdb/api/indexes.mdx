## B-tree Indexes

Create indexes on frequently queried fields for O(log n) lookups instead of full scans.

```typescript
const col = await db.collection("tasks");

// Create an index
col.createIndex("status");

// Queries on "status" now use the index automatically
col.find({ filter: { status: "active" } });     // index used
col.find({ filter: { status: { $gte: "a" } } }); // range queries too
col.count({ status: "active" });                 // O(1) with index

// List and drop
col.listIndexes();      // → ["status"]
col.dropIndex("status");
```

### Range Query Support

Indexed fields support `$gt`, `$gte`, `$lt`, `$lte` operators:

```typescript
col.createIndex("score");
col.find({ filter: { score: { $gte: 10, $lte: 90 } } }); // uses index
```

### Count Fast Path

<Aside type="tip">
  When a filter targets a single indexed field and no records have TTL, `count()` returns the index size directly — no record fetches needed.
</Aside>

## Composite Indexes

For compound queries on multiple fields:

```typescript
col.createCompositeIndex(["status", "priority"]);

// All-equality lookups use the composite index
col.find({ filter: { status: "active", priority: "H" } });

// Management
col.listCompositeIndexes();  // → [["status", "priority"]]
col.dropCompositeIndex(["status", "priority"]);
```

## Bloom Filters

Probabilistic existence checks — fast "definitely not" answers:

```typescript
col.createBloomFilter("email", 50000); // expected items

col.mightHave("email", "alice@example.com"); // true or false
// true = maybe exists, false = definitely doesn't exist
```

## Index Suggestions

AgentDB tracks which fields are queried most often:

```typescript
// After many queries...
col.suggestIndexes(100);
// → [{ field: "status", count: 342 }, { field: "role", count: 156 }]
```

## How Indexes Work

- **Sorted array** with binary search — O(log n) equality and range lookups
- **Reverse map** (id → key) for O(1) removal without knowing the indexed value
- **Incrementally maintained** on every insert, update, upsert, delete
- **Rebuilt on tail/watch** only for affected records (not full scan)
- **Automatically used** by `find()`, `count()`, `update()`, and `remove()`

## Text Index (BM25)

Fields marked `searchable: true` in [defineSchema](/agentdb/api/schemas/) are added to a per-collection BM25 inverted index. The index powers `bm25Search`, the lexical arm of `hybridSearch`, and the `db_bm25_search` MCP tool. See [Search](/agentdb/api/vector-search/) for the search reference.

- **Unicode tokenizer** — `[\p{L}\p{M}\p{N}]+` matches accented Latin, CJK, Hangul, etc. Emoji excluded. No NFC/NFD normalisation.
- **BM25 stats persisted** in disk-mode (TextIndex v2 JSON); v1 indexes upgrade lazily on first save.
- **Memory accounting** — `TextIndex.estimatedBytes()` registers with the memory monitor; surfaced as `textIndexBytes` in `Collection.stats()` and `db_stats`.
- **Same-session writes** are searchable immediately via WAL fallback in disk mode.
- **Size cap** — disk-mode text index is capped at 256 MB on load (~25–30K docs); past this, BM25 throws `IndexFileTooLargeError`.

## HNSW Vector Index

`defineSchema` collections with an embedding provider build an HNSW (Hierarchical Navigable Small World) index for approximate nearest neighbor search. Disk-mode HNSW is rebuilt from stored vectors on reopen, with auto-detected dimensionality (relevant for providers like Ollama that report `dimensions=0` until first call). See [Search](/agentdb/api/vector-search/).
