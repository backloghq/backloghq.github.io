## B-tree Indexes

Create indexes on frequently queried fields for O(log n) lookups instead of full scans.

```typescript
const col = await db.collection("tasks");

// Create an index
col.createIndex("status");

// Queries on "status" now use the index automatically
col.find({ filter: { status: "active" } });     // index used
col.find({ filter: { status: { $gte: "a" } } }); // range queries too
col.count({ status: "active" });                 // O(1) with index

// List and drop
col.listIndexes();      // → ["status"]
col.dropIndex("status");
```

### Range Query Support

Indexed fields support `$gt`, `$gte`, `$lt`, `$lte` operators:

```typescript
col.createIndex("score");
col.find({ filter: { score: { $gte: 10, $lte: 90 } } }); // uses index
```

### Count Fast Path

<Aside type="tip">
  When a filter targets a single indexed field and no records have TTL, `count()` returns the index size directly — no record fetches needed.
</Aside>

## Composite Indexes

For compound queries on multiple fields:

```typescript
await col.createCompositeIndex(["status", "priority"]);

// All-equality lookups use the composite index
col.find({ filter: { status: "active", priority: "H" } });

// Management
col.listCompositeIndexes();  // → [["status", "priority"]]
col.dropCompositeIndex(["status", "priority"]);
```

<Aside type="note">
  Disk-mode persistence (v2.2) — composite indexes now persist to `<dir>/indexes/composite-{fields__joined}.json` on close and lazy-load on reopen, so the v2.1.1 O(N) disk-scan rebuild only runs when no JSON file exists yet (first open after upgrade) or version mismatch is detected. Field-set validation guards against the rare filename-collision case where two field arrays join to the same on-disk name. Corrupt JSON or non-ENOENT load failures emit a diagnostic warning before falling back to disk-scan rebuild.
</Aside>

## Bloom Filters

Probabilistic existence checks — fast "definitely not" answers:

```typescript
await col.createBloomFilter("email", 50000); // expected items

col.mightHave("email", "alice@example.com"); // true or false
// true = maybe exists, false = definitely doesn't exist
```

<Aside type="tip">
  Auto-consulted by the query planner (v2.2) — `find()`, `count()`, `update()`, and `delete()` short-circuit to zero results when a bloom filter on the queried field returns `mightHave=false` for an equality predicate (`{ field: value }`, `{ field: { $eq: value } }`, all-absent `{ field: { $in: [...] } }`). B-tree, composite, and array indexes win when both exist. False positives fall through to a normal scan — correct by design.
</Aside>

<Aside type="note">
  Disk-mode persistence (v2.2) — bloom filters persist to `<dir>/indexes/bloom-{field}.json` on close and lazy-load on reopen; the v2.1.1 disk-iter populate path remains as fallback for first open after upgrade or version mismatch. Post-creation inserts/updates/deletes now correctly update the bloom filter (was silently stale pre-v2.2).
</Aside>

## Index Suggestions

AgentDB tracks which fields are queried most often:

```typescript
// After many queries...
col.suggestIndexes(100);
// → [{ field: "status", count: 342 }, { field: "role", count: 156 }]
```

## How Indexes Work

- **Sorted array** with binary search — O(log n) equality and range lookups
- **Reverse map** (id → key) for O(1) removal without knowing the indexed value
- **Incrementally maintained** on every insert, update, upsert, delete
- **Rebuilt on tail/watch** only for affected records (not full scan)
- **Automatically used** by `find()`, `count()`, `update()`, and `remove()`

## Text Index (BM25)

Fields marked `searchable: true` in [defineSchema](/agentdb/api/schemas/) are added to a per-collection BM25 index. The index powers `bm25Search`, the lexical arm of `hybridSearch`, and the `db_bm25_search` MCP tool. See [Search](/agentdb/api/vector-search/) for the search reference.

As of v2.0 the BM25 backend is [`@backloghq/termlog`](https://github.com/backloghq/termlog) — a segment-based LSM full-text index persisted at `<dir>/text/`.

- **Unicode tokenizer** — `[\p{L}\p{M}\p{N}]+` with NFC normalisation; matches accented Latin, CJK, Hangul, etc. Emoji excluded.
- **k1 / b tunable** — set via `defineSchema({ bm25: { k1, b } })` or `CollectionOptions.bm25` (defaults `k1=1.2`, `b=0.75`).
- **No size cap** — segments are size-tiered and compacted; collections grow without an artificial document ceiling.
- **S3-aware** — when AgentDB runs on S3 (via `@backloghq/opslog-s3`), termlog transparently uses [`@backloghq/termlog-s3`](https://github.com/backloghq/termlog-s3) (optional peer dep) and stores segments under the same bucket prefix.
- **Same-session writes** are searchable immediately — `bm25Search`, `$text`, and `search()` flush the write buffer before querying.
- **v1.4 → v2.0 migration** — opening a collection with a legacy `indexes/text-index.json` blob throws `LegacyTextIndexError`. Recover via `await db.rebuildTextIndex(name)` (or the `db_rebuild_text_index` MCP tool); see the [migration guide](/agentdb/getting-started/#v14-→-v20-migration).

## HNSW Vector Index

`defineSchema` collections with an embedding provider build an HNSW (Hierarchical Navigable Small World) index for approximate nearest neighbor search.

- **Graph persisted to disk (v2.1.1)** — disk-mode collections write the HNSW topology to `<collectionDir>/hnsw/graph.bin` on `close()` and load it on reopen. Vectors hydrate O(N) from quantized `_embedding` fields already in Parquet/JSONL. Falls back to a full rebuild on magic/version mismatch or post-load consistency failure.
- **Auto-detected dimensionality** for providers like Ollama that report `dimensions=0` until first call.
- **Tunable via `CollectionOptions.hnsw` and `AgentDBOptions.hnsw` (v2.1+)** — `M` (default `16`), `efConstruction` (default `200`), `efSearch` (default `50`), `maxLevel` (default `max(16, floor(log(1e6)/log(M)))`), `seed` (optional, deterministic layer assignment).

```typescript
const docs = await db.collection(schema, {
  hnsw: { M: 32, efConstruction: 400, efSearch: 100, seed: 42 },
});
```

See [Search](/agentdb/api/vector-search/) for the search reference and [Configuration](/agentdb/guides/configuration/) for the env-var equivalents.
